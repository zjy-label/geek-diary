# 大模型时代的流式响应与智能拦截架构方案

## 1. 为什么传统请求模式在 AI 时代失效？
传统前端基于 Request-Response 的 JSON 交互，在遇到大模型（LLM）动辄 3-10 秒的生成延迟时，会导致极差的用户体验。AI 时代的 UX 核心在于**消除等待的焦虑感**。

## 2. 流式响应 (Streaming Response) 的前端架构
流式响应的本质是降低 TTFT（Time To First Token），边生成边渲染。

### 核心技术点与面试高频考点
- **通信协议**：放弃传统 Ajax/Axios，转向使用 SSE (Server-Sent Events) 或原生的 `fetch` 结合 `ReadableStream` API。
- **渲染性能优化 (React 场景)**：
  - 高频触发的文本流会导致严重的 Re-render 灾难。
  - **解决方案**：避免直接将流式文本放入组件顶层的 `useState`。可以使用 `useRef` 缓存完整字符串，结合特定的子组件订阅更新，或者直接操作 DOM node 的 `textContent` 规避 React 虚拟 DOM 的比对开销。
- **不完整 Markdown 容错渲染**：
  - 流式到达的数据块经常会截断 Markdown 语法（如只到达了一半的代码块记号 ` ``` `）。
  - **解决方案**：引入基于 AST 的容错解析器（如 `remark`/`rehype` 体系），在渲染层容忍未闭合标签。
- **智能滚动接管**：
  - 当内容不断追加时，需自动 `scrollToBottom`。但必须监听用户的 `wheel` 或 `touchmove` 事件，一旦用户上滑查看历史，立即暂停自动滚动，避免抢夺控制权。

## 3. 智能拦截 (Intelligent Interception)：把 AI 藏在操作流中
未来的 AI 产品将不再依赖“对话框”形态，而是转为伴随式、前置式的隐形干预。

### 交互形态与前端实现
- **幽灵文本 (Ghost Text) 补全**：
  - 类似 GitHub Copilot，在用户输入时进行拦截补全。
  - **前端实现**：利用 CSS 的 `position: absolute`、`color: transparent` 技巧叠加输入层，或者使用 Shadow DOM 隔离样式的灰色建议文本。结合节流防抖（Debounce）发送请求，用户按 `Tab` 键采纳。
- **前置意图校验 (Pre-validation)**：
  - 在表单提交前，由极小参数量模型进行语义逻辑拦截。
  - **区别于传统正则**：正则只能校验格式（如邮箱格式），AI 拦截能校验语义冲突（如“结束日期早于开始日期”，或者“预算与描述范围不符”）。

## 4. 商业启发 (给小团队的建议)
放弃做同质化的“全屏对话型”应用。切入痛点场景（如客户服务、邮件回复、复杂表单填写），利用 Chrome Extension 或注入脚本，在用户原有的操作界面上实现**智能拦截与流式补全**。这样的产品形态极轻，且一旦切中痛点，用户黏性极高。
